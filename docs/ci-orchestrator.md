# Haskell CI Orchestrator Integration

## Overview

The file `.github/workflows/ci.yml` is **generated by the Haskell CI orchestrator** and must not be edited manually. The orchestrator manages CI configurations across multiple repos.

## Files

| File | Managed By | Editable? |
|------|-----------|-----------|
| `.github/workflows/ci.yml` | haskell-ci-orchestrator | **NO** — regenerated on push |
| `.github/workflows/dart-ci.yml` | manual | YES — Dart/web-specific CI |

## How It Works

1. The orchestrator runs on a schedule and on push to the orchestrator repo
2. It reads project configs and generates CI YAML for each managed project
3. Generated files are committed directly to each project's `.github/workflows/`
4. The `# Generated by haskell-ci-orchestrator — DO NOT EDIT MANUALLY` header marks managed files

## Dart CI

Since the Dart web server was added after the orchestrator was configured, Dart-specific CI lives in a separate `dart-ci.yml` file that only triggers on Dart/web file changes. This avoids conflicts with the orchestrator-generated `ci.yml`.

To integrate Dart CI into the orchestrator:
1. Update the orchestrator's project config to include Dart lint/test jobs
2. Add `lib/`, `bin/`, `test/dart/`, `pubspec.*` to the orchestrator's path triggers
3. After orchestrator regeneration, the separate `dart-ci.yml` can be removed

## Notification

When the orchestrator regenerates CI for aichat, it:
- Creates a commit with the updated `ci.yml`
- The commit message contains `[ci-orchestrator]` for easy filtering
- Monitor with: `git log --oneline --grep='ci-orchestrator'`
